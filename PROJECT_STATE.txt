--------------------------------------------------------------------------------
1. PROJECT SUMMARY
--------------------------------------------------------------------------------

A production-grade real-time streaming data pipeline that ingests live BSE
(Bombay Stock Exchange) stock prices via the Alpha Vantage API, streams them
through Apache Kafka, processes them with PySpark Structured Streaming, and
writes to two simultaneous sinks: AWS S3 (data lake) and Snowflake (data
warehouse). The full stack is containerized with Docker Compose.

This is Part 2 of a two-project data engineering portfolio. Part 1 (batch ELT)
lives at: https://github.com/Samik7hos0/market-pipeline — same BSE domain,
daily schedule, dbt transformations, Airflow orchestration.

4. DATA FLOW
--------------------------------------------------------------------------------

Alpha Vantage API (REST, GLOBAL_QUOTE)
  ↓  (every 60s, 13s sleep between each stock to respect 5 calls/min rate limit)
Python Producer (producer/stock_producer.py)
  ↓  (JSON messages, keyed by stock symbol → deterministic partition routing)
Kafka Topic: "stock-prices" (4 partitions, 1 per stock)
  ↓  (micro-batch every 30 seconds)
Spark Structured Streaming Consumer (consumer/spark_consumer.py)
  ↓  foreachBatch — dual-write in single processing step
  ↓                                 ↓
AWS S3                         Snowflake
s3a://<bucket>/                STREAMING.STOCK_TICKS
  streaming/stock_prices/      (append mode, structured rows)
  (raw immutable JSON)

11. KEY ENGINEERING DECISIONS
--------------------------------------------------------------------------------

1. KAFKA PARTITIONING BY SYMBOL KEY
   Symbol used as Kafka message key → hash → deterministic partition assignment.
   Preserves per-symbol time-series ordering for downstream consumers.

2. foreachBatch DUAL-SINK PATTERN
   Single Spark query writes to S3 and Snowflake in the same micro-batch call.
   Each sink wrapped in independent try/except — partial failure is tolerated.
   Avoids double Kafka reads or double checkpoints.

3. EXACTLY-ONCE SEMANTICS VIA CHECKPOINTING
   Spark writes Kafka offsets to checkpoint directory after each successful batch.
   On restart, resumes from last committed offset. No data loss, no duplicates.
   Checkpoint path: configurable via SPARK_CHECKPOINT env var.

4. DUAL-SINK DATA LAKEHOUSE PATTERN
   S3: raw immutable JSON — source of truth, reprocessable at any time.
   Snowflake: structured queryable rows for analytics.
   Follows lakehouse pattern used in production fintech (Razorpay, CRED, Groww).

5. RATE-LIMIT-AWARE PRODUCER
   Alpha Vantage free tier: 5 calls/min.
   Producer sleeps 13s between each of 4 stocks → full cycle ~52s + 60s sleep.
   Production upgrade path: websocket feed or premium API tier.

14. KNOWN ISSUES / GAPS
--------------------------------------------------------------------------------

1. utils/snowflake_writer.py is empty — only 1 line, placeholder, not used.
   The Snowflake write logic is inline in spark_consumer.py's write_batch().

2. README says 5 stocks (including WIPRO.BSE) but producer code only tracks 4.
   WIPRO.BSE is missing from the STOCKS dict in stock_producer.py.

3. KAFKA_ADVERTISED_LISTENERS in docker-compose uses PLAINTEXT://localhost:9092
   but the producer container connects via kafka:9092 (Docker internal network).
   This can cause connection issues when producer runs inside Docker — the
   advertised listener should be PLAINTEXT://kafka:9092 for internal networking.

4. Spark checkpoint path is hardcoded as Windows path fallback
   (C:/tmp/spark-checkpoints/stock-stream). The Docker consumer container
   overrides this via the SPARK_CHECKPOINT env var to /tmp/spark-checkpoints/
   stock-stream, which is correct for Linux containers.

5. No tests exist (unit or integration).

6. No monitoring or alerting configured.
